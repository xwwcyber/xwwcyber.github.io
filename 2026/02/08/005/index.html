git分支相关技巧

编辑原文
2026-02-08 tips 1.5k 字
git

1. 什么是 Git 分支？

在 Git 中，分支本质上只是一个指向特定提交（Commit）的可移动指针。

  • 默认分支：Git 初始化时默认创建一个分支，通常叫 master 或 main。
  • HEAD 指针：Git 使用一个名为 HEAD 的特殊指针来知道当前你正在哪个分支上工作。
  • 轻量级：创建分支几乎是瞬间完成的，因为它只是创建了一个 40 字节的文件（即指针），而不是复制整个项目的文件。

2. 常用分支命令清单

这是你需要掌握的日常操作命令：

查看与创建

  • 查看本地分支：git branch (当前分支前会有 * 号)
  • 查看所有分支（含远程）：git branch -a
  • 创建新分支：git branch <分支名> (仅创建，不切换)

切换分支

  • 切换分支（旧命令）：git checkout <分支名>
  • 切换分支（新命令，推荐）：git switch <分支名>
  • 创建并立即切换：git checkout -b <分支名> 或 git switch -c <分支名>

合并与删除

  • 合并分支：git merge <分支名> (将指定分支合并到当前分支)
  • 删除已合并分支：git branch -d <分支名>
  • 强制删除分支（慎用）：git branch -D <分支名> (当分支未合并但你想丢弃它时)

3. 解决冲突

当两个分支修改了同一个文件的同一行代码，Git 无法自动决定使用哪一个，就会产生冲突。

解决步骤：

  1. Git 会暂停合并，并告诉你是哪个文件冲突。

  2. 打开冲突文件，你会看到类似标记：

    <<<<<<< HEAD
    var name = "Alice"; (当前分支的代码)
    =======
    var name = "Bob";   (要合并进来的分支代码)
    >>>>>>> feature-login

  3. 手动编辑：删除标记符号，保留你想要的代码（或两者组合）。

  4. 标记完成：git add <文件名>

  5. 提交：git commit (完成合并)

4.使用场景

为了让你更直观地掌握，我们将模拟一个最真实的日常开发场景。

假设你正在做一个项目，你需要开发一个新功能（比如“登录页面”），然后将其合并回主分支。

请打开你的终端（Terminal）或 Git Bash，按以下步骤操作：

场景一：标准的“新建-开发-合并”流程（最常用）

这是最顺利的情况，你从主分支切出去，写完代码，然后合并回来。

1. 确保主分支是最新的

在开始任何工作前，先回到主分支并拉取最新代码，防止落后。

git checkout main      # 切换回主分支
git pull origin main   # 从远程拉取最新代码（如果是多人协作）

2. 创建并切换到新分支

我们要开发登录功能，所以给分支起个好名字。

git checkout -b feat/login-page
# 这条命令等同于：git branch feat/login-page + git checkout feat/login-page

此时，你的终端提示符应该已经显示你处于 feat/login-page 分支下。

3. 写代码并提交

假设你新建了一个 login.html 文件。

# ...在此处进行你的代码编写...

git add .                          # 将修改添加到暂存区
git commit -m "完成登录页面的基础布局"  # 提交修改到当前分支

4. 合并回主分支

功能开发完了，我们要把它合进 main。

git checkout main         # 第一步：必须先切回目标分支（即 main）
git merge feat/login-page # 第二步：把 feat/login-page 合并进当前分支

如果一切顺利，你会看到 “Fast-forward” 的提示，代表合并成功。

5. 删除功能分支（可选）

既然代码已经合进去了，这个旧分支就可以清理掉了。

git branch -d feat/login-page

场景二：解决代码冲突（必须掌握）

这是新手最怕的环节。假设你和同事都修改了 README.md 的同一行，合并时 Git 会报错：CONFLICT (content): Merge conflict in README.md。

1. 发现冲突

当你执行 git merge 后，终端提示 Conflict，此时 Git 会暂停合并过程。

2. 打开文件手动修复

用代码编辑器（VS Code 等）打开冲突的文件 README.md。你会看到类似这样的标记：

<<<<<<< HEAD
这是主分支原本的内容：项目简介 V1
=======
这是你在新分支修改的内容：项目简介 V2 (更详细)
>>>>>>> feat/update-readme

你的任务是：

  1. 做选择：决定保留哪段代码（或者两段都保留，或者重写）。
  2. 删标记：删除 <<<<<<<、=======、>>>>>>> 这些符号。

修改后的样子可能长这样：

这是你在新分支修改的内容：项目简介 V2 (更详细)

3. 提交解决后的结果

修改完所有冲突文件后，告诉 Git 你解决好了。

git add README.md    # 标记该文件冲突已解决
git commit           # 不需要写 -m，Git 会自动生成一个"Merge branch..."的提交信息

此时，合并完成。

场景三：将本地分支推送到远程（GitHub/GitLab）

当你第一次在本地创建了分支，远程仓库（GitHub）是不知道的。你需要把它“推”上去。

# 假设你还在 feat/login-page 分支上
git push -u origin feat/login-page

  • 注意： 第一次推送必须加 -u（即 --set-upstream），这样以后你在这个分支只需要输入 git push 或 git pull，Git 就知道它是对应远程的哪个分支了。

总结：命令速查表

目的 命令
我要开始新工作 git checkout -b <新分支名>
我要保存进度 git add . 然后 git commit -m "备注"
我要把代码合进主干 先 git checkout main，再 git merge <分支名>
我要把分支传给同事 git push -u origin <分支名>
我写乱了，想放弃修改 git checkout . (危险操作：会丢弃所有未提交的修改)
